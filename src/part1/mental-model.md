# Mental Model: EVM → LEZ

This chapter is written for developers who know Solidity and the EVM. If that's you, you already understand the hard parts — programmable state, access control, upgradability tradeoffs, and the call model. You just need the translation layer.

The concepts map cleanly. The syntax is different. The one genuinely new idea is how state is managed. Everything else follows from that.

---

## Concept Map at a Glance

| EVM / Solidity | LEZ / SPEL | Notes |
|---|---|---|
| Contract | Program | Rust crate with `#[lez_program]` |
| Storage slots | Accounts (external state) | State lives in accounts, not inside the program |
| `msg.sender` | Signer account | An account with `#[account(signer)]` |
| Contract address | Program ID (RISC Zero ImageID) | Hash of the zkVM guest binary |
| Solidity | Rust + SPEL macros | `#[lez_program]`, `#[instruction]` |
| EVM bytecode | RISC Zero zkVM guest binary | Generates a ZK proof on execution |
| `constructor` | `init` instruction with `#[account(init)]` | Account must have default/empty data |
| Transaction calldata | Instruction enum (Borsh-serialized) | Typed, not raw bytes |
| Events/logs | No direct equivalent yet | — |
| `mapping(k => v)` | PDA-derived accounts | Seeds → deterministic address |
| Function modifiers | Account constraints | `signer`, `owner`, `init`, `mut` |
| `delegatecall` | CPI (ChainedCall) | Cross-Program Invocation |
| ABI | IDL | JSON interface, generated by SPEL |

## Contracts → Programs

### In Solidity

A contract is the unit of deployment. It owns its own storage, has its own address, and holds both the logic and the state:

```solidity
contract Counter {
    uint256 public count;
    address public owner;

    constructor(address _owner) {
        owner = _owner;
        count = 0;
    }

    function increment() external {
        require(msg.sender == owner, "unauthorized");
        count += 1;
    }
}
```

The contract address identifies this specific instance. Storage slot `0` is `count`, slot `1` is `owner`. The code and the data share an identity.

### In LEZ

A program is a Rust crate compiled to `riscv32im` (RISC-V 32-bit) and loaded into the RISC Zero zkVM. The program is identified by its **ImageID** — the hash of the compiled guest binary. Multiple independent state accounts can be owned by the same program.

```rust
use spel::prelude::*;

#[lez_program]
mod counter {
    #[derive(Default, BorshSerialize, BorshDeserialize)]
    pub struct CounterState {
        pub count: u64,
        pub owner: [u8; 32],
    }

    #[instruction]
    pub fn increment(
        #[account(mut)] counter: AccountWithMetadata<CounterState>,
        #[account(signer)] authority: AccountWithMetadata<()>,
    ) -> ProgramResult {
        require!(authority.id() == counter.data.owner, "unauthorized");
        counter.data.count += 1;
        Ok(())
    }
}
```

Notice: there are no member variables. `CounterState` is a struct that lives in an *account*, not inside the program module. The `increment` function receives that account as a parameter.

> **🔄 Coming from Solidity?** A LEZ program is not a deployed instance — it's more like a class definition. The "instance" is the account. You can have thousands of `CounterState` accounts all governed by the same program ImageID.

## Storage → Accounts

This is the most important conceptual shift in the entire chapter.

### In Solidity

State is implicit. You declare state variables at the contract level, and they're magically persisted between calls:

```solidity
contract Token {
    mapping(address => uint256) public balances;
    uint256 public totalSupply;

    function transfer(address to, uint256 amount) external {
        require(balances[msg.sender] >= amount);
        balances[msg.sender] -= amount;
        balances[to] += amount;
    }
}
```

You never think about "passing in" the `balances` mapping. It's just there.

### In LEZ

State is explicit. Every piece of state your instruction touches must be passed in as an account parameter. The program doesn't own a storage area — accounts do.

An account is a structured object with these fields:

- **address**: a 32-byte identifier (user-chosen or PDA-derived)
- **balance**: native token balance
- **nonce**: replay protection counter
- **owner**: the ImageID of the program authorized to write to it
- **data**: a Borsh-serialized struct (your `CounterState`, `TokenAccount`, etc.)

In SPEL, accounts are typed via `AccountWithMetadata<T>`:

```rust
// T is your data struct — deserialized from the account's raw bytes
let count: u64 = counter.data.count;

// Metadata (address, balance, nonce, owner) is on the wrapper
let addr: [u8; 32] = counter.id();
let balance: u64 = counter.lamports();
```

When you mark an account `#[account(mut)]`, the SPEL runtime serializes and writes back the modified data at the end of the instruction. If you don't mark it `mut`, the account is read-only and writes are rejected.

> **⚠️ Warning:** The `#[account(mut)]` attribute is a constraint checked at runtime by the SPEL framework. If you forget it on an account you modify, the instruction will fail — even if the Rust compiler accepts the code.

## `msg.sender` → Signer Account

### In Solidity

`msg.sender` is a global available in every function call. It's the address of the caller, implicitly authenticated by the transaction signature:

```solidity
require(msg.sender == owner, "not the owner");
```

### In LEZ

There is no `msg.sender` global. Instead, you declare a signer account as an explicit parameter with `#[account(signer)]`. The SPEL runtime verifies that the corresponding private key signed the transaction before your function body runs.

```rust
#[instruction]
pub fn increment(
    #[account(mut)] counter: AccountWithMetadata<CounterState>,
    #[account(signer)] authority: AccountWithMetadata<()>,
) -> ProgramResult {
    require!(authority.id() == counter.data.owner, "unauthorized");
    counter.data.count += 1;
    Ok(())
}
```

The `#[account(signer)]` constraint means: "the transaction must include a valid signature from the private key corresponding to this account's address." The `.id()` call retrieves that address, which you can then compare against whatever is stored in your state.

> **🔄 Coming from Solidity?** `msg.sender` is not a free global in LEZ — it's a first-class account you explicitly declare. This makes the authorization model visible in the function signature rather than hidden in the body. It also means you can have multiple signers in a single instruction.

## Constructor → `init` Instruction

### In Solidity

The constructor runs once at deployment, setting up initial state:

```solidity
constructor(address _owner) {
    owner = _owner;
    count = 0;
}
```

### In LEZ

There is no constructor. Instead, you write an `initialize` (or `init`) instruction that uses `#[account(init)]`. This constraint verifies that the target account's data is currently default/empty — ensuring initialization happens exactly once.

```rust
#[instruction]
pub fn initialize(
    #[account(init, mut)] state: AccountWithMetadata<CounterState>,
    #[account(signer)] authority: AccountWithMetadata<()>,
) -> ProgramResult {
    state.data.owner = authority.id();
    state.data.count = 0;
    Ok(())
}
```

The `init` constraint checks that `state.data` equals `CounterState::default()` before your code runs. If the account already has non-default data, the instruction fails. This prevents re-initialization attacks.

The `mut` constraint is also required here — you can't write to an account without it.

> **⚠️ Warning:** `#[account(init)]` checks that the data equals `Default::default()`, not that the account is new. Make sure your state struct derives `Default` and that the default value is a meaningful "uninitialized" sentinel.

## Mappings → PDA-Derived Accounts

### In Solidity

Mappings are a built-in storage structure that gives you a key→value store:

```solidity
mapping(address => uint256) public balances;

function getBalance(address user) external view returns (uint256) {
    return balances[user];
}
```

The slot for `balances[user]` is computed deterministically from the mapping's slot and the key — but that's invisible to you.

### In LEZ

There are no mappings. Instead, you use **PDA-derived accounts** (Program Derived Addresses). A PDA is an account whose address is derived from a set of seeds plus the program's ImageID. If you know the seeds and the program ID, you can always compute the address — it's deterministic.

In practice, you define the seeds for a per-user balance account like this:

```rust
// Seeds might be: [literal("balance"), account("user")]
// The address is: hash(program_image_id || "balance" || user_address)
```

At the call site, the client computes the PDA address from the seeds and passes it as an account parameter. The program can re-derive and verify the address on-chain if needed.

> **🔄 Coming from Solidity?** The mental model shift is: instead of `balances[user]`, you have a separate *account* per user, whose address is deterministically derived from `user`. The "lookup" becomes "compute the address from the key, then pass that account to the instruction."

This is more verbose than a mapping but more powerful: each PDA account can hold arbitrary structured data, have its own balance, and be independently verified.

## Function Modifiers → Account Constraints

### In Solidity

Modifiers let you attach conditions to functions declaratively:

```solidity
modifier onlyOwner() {
    require(msg.sender == owner, "not the owner");
    _;
}

function withdraw() external onlyOwner {
    payable(msg.sender).transfer(address(this).balance);
}
```

### In LEZ

Account constraints on `#[instruction]` parameters serve the same role. They're checked by the SPEL runtime before your function body executes:

```rust
#[instruction]
pub fn withdraw(
    #[account(mut)] vault: AccountWithMetadata<VaultState>,
    #[account(signer)] authority: AccountWithMetadata<()>,
    #[account(owner = PROGRAM_ID)] config: AccountWithMetadata<ConfigState>,
) -> ProgramResult {
    require!(authority.id() == vault.data.owner, "not the owner");
    // ...
    Ok(())
}
```

Available constraints:

| Constraint | Effect |
|---|---|
| `signer` | Transaction must include a valid signature for this account |
| `mut` | Account is writeable; SPEL writes back changes at end |
| `init` | Account data must equal `Default::default()` |
| `owner = <IMAGE_ID>` | Account's `owner` field must equal the given ImageID |

You can combine constraints: `#[account(init, mut)]` is common for initialization.

> **🔄 Coming from Solidity?** Modifiers in Solidity run arbitrary code. Account constraints in LEZ are a closed set of declarative checks. More complex conditions (like "this authority must match the stored owner") still need to be written as `require!()` statements in your function body.

## `delegatecall` → Cross-Program Invocation (CPI)

### In Solidity

`delegatecall` lets one contract call another's code in the caller's storage context. It's powerful but dangerous — it's the mechanism behind most proxy upgradability patterns:

```solidity
(bool success, bytes memory data) = implementation.delegatecall(
    abi.encodeWithSignature("doSomething(uint256)", 42)
);
```

### In LEZ

LEZ supports **Cross-Program Invocation (CPI)**, implemented via `ChainedCall`. A program can invoke another program's instruction within the same transaction, passing accounts along.

CPI in LEZ does not have `delegatecall` semantics (executing foreign code in your own context). It is a proper cross-program call: the target program runs in its own zkVM context with its own ImageID constraints.

```rust
// Calling another program from within an instruction
let result = ChainedCall::new(OTHER_PROGRAM_IMAGE_ID)
    .instruction(OtherInstruction::DoSomething { value: 42 })
    .account(some_account)
    .invoke()?;
```

> **⚠️ Warning:** CPI in LEZ has composability implications for ZK proofs. Each program call produces its own proof, and the outer program's proof must verify the inner one. This is handled by the SPEL runtime and RISC Zero's proof composition — but be aware it adds proving time proportional to the call depth.

## ABI → IDL

### In Solidity

The ABI (Application Binary Interface) is the JSON schema describing a contract's external interface — function signatures, argument types, event definitions. It's generated by the Solidity compiler and used by client libraries like `ethers.js`.

### In LEZ

The equivalent is the **IDL** (Interface Definition Language), a JSON file generated by SPEL when you build your program. It describes:

- All `#[instruction]` functions and their parameter types
- All account structs and their fields
- The program's ImageID

`lez-cli` uses the IDL to generate typed client code. The `make cli` target in scaffolded projects reads the IDL to know how to serialize instruction calls.

```bash
# After build, your IDL is at:
target/lez/my_program.idl.json
```

## The Privacy Dimension

Everything above is a syntax/API translation. Privacy is the one dimension that has no EVM equivalent at all.

### In EVM

All state is public. Every storage slot is readable by anyone. Private data requires off-chain storage, trusted servers, or external ZK systems that are disconnected from your contract logic.

Existing solutions:
- **Tornado Cash**: breaks the link between deposit and withdrawal addresses — not a general-purpose privacy layer
- **Aztec**: a separate chain with its own ZK circuit language (Noir) — general-purpose but requires a full rewrite

### In LEZ

Every program natively supports private transactions without any code changes.

A **public transaction** in LEZ:
- Inputs and outputs (account addresses, data changes) are visible to `lssa` and the ledger
- Works like a normal smart contract call

A **private transaction** in LEZ:
- Inputs and outputs are hidden using commitments and nullifiers (the same cryptographic primitives as Zcash shielded pools)
- `lssa` verifies the ZK proof and updates a commitment tree — it never sees the plaintext account data
- The caller's wallet holds the decryption key for their own accounts

Both modes use **the same compiled program binary**. You do not write a public version and a private version. The privacy circuit wraps your program at the protocol level, applied by `lssa` depending on the transaction type.

From the developer's perspective:

```bash
# Public call
make cli ARGS="increment --counter <addr>"

# Private call — same program, different transaction envelope
make cli ARGS="increment --counter <addr> --private"
```

Your Rust code does not change. The privacy machinery is a layer beneath your program, not a feature you implement.

> **💡 Tip:** Private transactions have higher proving cost because the privacy circuit adds zkVM work on top of your program logic. For performance-sensitive applications, design so that only the operations that genuinely need privacy use the `--private` path.

## The Fundamental Mental Shift

> **🔄 Coming from Solidity?** The hardest mental shift isn't the syntax — it's that **state doesn't live inside your program**. Your program is stateless code that accepts accounts as inputs and returns updated accounts as outputs. Think of it less like a class with member variables, and more like a pure function over an account database.

A concrete way to internalize this:

```
// Solidity mental model
contract.increment()         // "tell the contract to update itself"

// LEZ mental model
increment(counter_account)   // "pass the state in, get updated state out"
```

The LEZ model has real advantages once you internalize it:

- **Composability**: any instruction can operate on any account that matches the type, without hard-coding addresses
- **Parallelism**: non-overlapping accounts can be processed in parallel (no shared mutable global state)
- **Auditability**: every account an instruction can touch is visible in its signature
- **Privacy**: account data can be hidden individually, without changing the program

The tradeoff is verbosity. You must explicitly list every account your instruction touches. For simple programs this is fine; for complex ones it requires careful architecture to avoid instruction signatures with ten account parameters.

## Summary Table

| Solidity pattern | LEZ equivalent | Key difference |
|---|---|---|
| `contract Foo { uint x; }` | `struct FooState { x: u64 }` in an account | State is external to the program |
| `msg.sender` | `#[account(signer)] authority` | Explicit, not a global |
| `constructor` | `initialize` instruction with `#[account(init)]` | One-time check via default data |
| `mapping(k => v)` | PDA-derived account per key | Address computed from seeds |
| `modifier onlyOwner` | `#[account(signer)]` + `require!` | Constraints are closed-set; logic is in body |
| `emit Transfer(...)` | No equivalent yet | Events not yet implemented |
| `delegatecall` | `ChainedCall::new(IMAGE_ID)` | No shared storage context |
| ABI JSON | IDL JSON | Same purpose, different format |
| All state is public | Public or private per transaction | Same binary, different envelope |
