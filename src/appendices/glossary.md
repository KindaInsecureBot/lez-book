# Appendix D: Glossary

An alphabetically ordered reference for LEZ terminology.

---

**Account** — The unit of state in LEZ. An account has an ID (address), data (borsh-serialized), balance, nonce, and `program_owner`. State lives in accounts, not in programs. Conceptually similar to an Ethereum account, but all accounts can hold arbitrary typed data.

**AccountPostState** — The return type for accounts from an instruction. Use `new_claimed()` for newly initialized accounts, `new()` for updated existing accounts, and `new_claimed_if_default()` for upsert semantics.

**AccountWithMetadata\<T\>** — The typed account wrapper used in SPEL instructions. `T` is your state struct. Access data with `.data()`, the account address with `.id()`, and create updated versions with `.with_data()`.

**Attestation** — A ZK proof that some fact about private state is true, without revealing the underlying state. Generated by running a SPEL program inside the privacy circuit. An attestation lets you prove "I own a tile" without revealing which tile.

**Borsh** — The serialization format used for all account data and instruction arguments in LEZ. Borsh (Binary Object Representation Serializer for Hashing) is deterministic and compact, which is required for ZK proof generation. See `borsh.io`.

**ChainedCall** — A CPI call appended to an instruction's output. The sequencer executes chained calls after the primary instruction completes successfully. Enables composable multi-step operations within a single transaction.

**Commitment** — The on-chain representation of a private account. Computed as `SHA256(npk || program_owner || balance || nonce || SHA256(data))`. Hides all account contents — the sequencer sees only the hash. Compare with Nullifier.

**CPI (Cross-Program Invocation)** — Calling another program's instruction from within your own program's instruction execution. The callee program must be registered with the sequencer.

**ELF** — The binary format for RISC Zero guest programs. RISC Zero compiles your Rust guest crate to a `riscv32im` ELF file. The hash of this ELF is the program's ImageID.

**Genesis Account** — An account funded and registered at chain genesis, or created by `wallet account new`. Genesis accounts can sign transactions. Contrast with PDA-derived accounts, which cannot sign (NSSA rule 7).

**Guest Binary** — The RISC Zero zkVM binary that contains your program logic. Every execution of the guest binary generates a ZK proof attesting to correct execution. The cryptographic hash of the guest binary is the program's ImageID.

**IDL (Interface Definition Language file)** — The JSON file describing your program's instructions, types, and account schemas. Equivalent to a Solidity ABI or a Protobuf definition. Generated by `make idl` (or `cargo run -p idl-gen`); never edit by hand.

**ImageID** — The cryptographic hash (SHA-256) of a RISC Zero zkVM guest binary (ELF file). Serves as the program's on-chain identifier — like a contract address in Ethereum. If any byte of the guest binary changes, the ImageID changes. Clients must be updated whenever a program is redeployed with a new ImageID.

**LEZ** — The smart contract platform documented in this book. Programs run inside the RISC Zero zkVM; every execution generates a ZK proof. Supports both public (transparent) and private (commitment-based) transactions.

**lez-cli** — The developer CLI for building, deploying, and calling LEZ programs. Primary subcommands: `init`, `call`, `deploy`, `account`, `pda`.

**lez-client-gen** — Tool that reads an IDL JSON file and generates typed Rust or C client code for interacting with a program. Regenerate whenever the IDL changes.

**lssa** — The LEZ sequencer and wallet implementation. `sequencer_runner` validates ZK proofs and processes transactions. `wallet` manages accounts and sends transactions. The two main binaries in the lssa repository.

**npk (Nullifier Public Key)** — `SHA256(nsk)`. Used to generate commitments for private accounts. Share your npk with senders so they can send you private funds without revealing your identity.

**nsk (Nullifier Secret Key)** — Your private key for nullifier derivation and private account spending. Never share this. Knowing nsk lets you spend private accounts, derive npk, and scan the chain for incoming private transactions.

**NSSA (Nondeterministic State Sequencer Architecture)** — The protocol architecture underlying LEZ. Defines the rules for how the sequencer processes transactions and validates state transitions. Key rule: rule 7 — only genesis accounts can be transaction signers; PDAs cannot sign.

**Nullifier** — The on-chain representation of a spent private account. Derived from nsk and the old commitment hash. Posted to the chain when a private account is consumed. Prevents double-spending by allowing the sequencer to reject transactions that reuse a nullifier.

**PDA (Program Derived Address)** — An account whose address is derived deterministically from seeds using `SHA256(program_id || seed_bytes...)`. Used to implement per-user state, per-entity records, and other mapping-like patterns. PDAs cannot sign transactions (NSSA rule 7).

**post\_states** — The list of `AccountPostState` values returned by an instruction via `LezOutput`. Must include ALL accounts that were passed in as parameters, even unmodified ones. Accounts not included in `post_states` are treated as deleted.

**Privacy Circuit** — The ZK circuit that wraps SPEL programs to enable privacy-preserving transactions. Handles encryption and decryption of account data, generates commitment and nullifier pairs, and verifies that the state transition was authorized by the correct nsk.

**Program** — A LEZ smart contract. A Rust crate compiled to `riscv32im`, identified by its ImageID. Programs are stateless — all state lives in accounts. Deploying a new program version changes its ImageID.

**RISC Zero** — The ZK proof system underlying LEZ. Programs compile to the `riscv32im` instruction set and run in the RISC Zero zkVM, which generates a proof of correct execution that anyone can verify without re-running the program.

**riscv32im** — The RISC-V 32-bit integer + multiply instruction set used as the RISC Zero guest target. Your LEZ program's guest crate compiles to this architecture. Some Rust crates don't support this target — notably `borsh_derive` (use `borsh` features = ["derive"] instead).

**rzup** — The RISC Zero toolchain manager. Installs the `riscv32im` compilation target and `cargo-risczero`. Use `curl -L https://risczero.com/install | bash && rzup install` to set up a fresh development environment.

**SPEL** — The macro framework for writing LEZ programs. The `#[lez_program]`, `#[instruction]`, and `#[account(...)]` attributes declare your program's interface and generate the dispatch logic, serialization, and PDA derivation code.

**Sequencer** — The node that receives transactions, verifies ZK proofs, and updates the chain state. In local development, `sequencer_runner --port 3040` runs a single-node sequencer backed by RocksDB.

**View Tag** — A 1-byte filter on private transactions that enables efficient wallet scanning. Computed from an ECDH shared secret between the sender and recipient. Allows the recipient's wallet to reject ~99.6% of irrelevant transactions without full decryption.

**Visibility Mask** — Per-account visibility setting in private transactions. Options: `public` (plaintext on chain, visible to all), `private-auth` (encrypted, only the owner can decrypt), `private-unauth` (encrypted, requires the view key to read).

**vpk (Viewing Public Key)** — A secp256k1 public key derived from nsk. Sharing your vpk lets someone scan the chain to see your incoming private transactions and balances, without being able to spend them. Useful for auditing or wallet recovery services.

**Workspace** — The Cargo workspace structure for a LEZ program. Typically contains `core/` (shared types), `methods/` (RISC Zero guest), `idl-gen/` (IDL generation binary), and optionally `cli-wrapper/` or `client/`. Generated by `lez-cli init`.

**zkVM** — Zero-Knowledge Virtual Machine. The RISC Zero execution environment where LEZ programs run. Executes the guest ELF, records the execution trace, and produces a ZK proof attesting that the output was produced by a correct execution of that specific binary on those specific inputs.
