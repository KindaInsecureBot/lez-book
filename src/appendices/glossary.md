# Appendix D: Glossary

An alphabetically ordered reference for LEZ terminology. EVM equivalents are noted where the analogy is reasonably direct.

---

## Account

The basic state container in LEZ. An account holds a blob of borsh-serialized data and is identified by its address (a 32-byte value). Unlike Solidity storage, accounts exist as first-class objects external to the program: a program does not own storage slots — it owns (and reads/writes) discrete account objects. Every instruction must return all accounts it received, even if unmodified.

> **🔄 Coming from Solidity?** The closest analog is a storage slot on a specific contract, but made external and portable. In Solidity, storage lives inside the contract. In LEZ, state lives in accounts that are passed to the program as arguments.

---

## AccountPostState

The return type for a modified (or passed-through) account in `LezOutput`. When an instruction finishes, it wraps each account in `AccountPostState` to hand the updated (or unchanged) data back to the sequencer. Constructed via `account.into_post_state(new_data_bytes)`. Every input account must be converted to an `AccountPostState` and included in the output, whether or not it was modified.

---

## AccountWithMetadata

The input type for accounts in instruction function signatures. Carries the account's address, owner program ID, and raw data bytes, along with metadata needed for constraint checking (signer flags, etc.). All `AccountWithMetadata` parameters must appear before scalar arguments in an `#[instruction]` function signature.

---

## Attestation

A ZK proof asserting a property about private state without revealing the state itself. In LEZ, attestations can prove claims like "this private account has a balance above threshold" or "this commitment was created by a valid program" — without exposing the actual balance or commitment contents. Attestations are the mechanism that makes private state usable in public contexts.

---

## borsh

Binary Object Representation Serialized as Hashes. A deterministic binary serialization format used throughout LEZ for account data and instruction arguments. borsh has a fixed encoding for each type, which is important for ZK proofs (the serialized bytes are what get committed and verified). The `borsh` crate provides the `BorshSerialize` and `BorshDeserialize` traits, but the derive macros (`borsh_derive`) cannot be used in guest code — manual implementations are required in the `riscv32im` guest environment.

> **🔄 Coming from Solidity?** borsh is to LEZ what ABI encoding is to Solidity — the canonical way data is packed for storage and transport. Unlike ABI encoding, borsh is compact and does not include type selectors.

---

## Commitment

A cryptographic hash of a private account's contents. Instead of storing private account data directly in the sequencer's public state, only the commitment (hash) is published. The actual account contents are known only to parties with the appropriate viewing key. Commitments are what the sequencer stores and indexes; the wallet scans commitments to find ones addressed to it.

> **🔄 Coming from Solidity?** Analogous to a commitment in a Pedersen or Poseidon hash-based scheme (as used in Tornado Cash or Zcash sapling). The commitment makes the existence of the private state publicly verifiable without revealing the state.

---

## CPI (Cross-Program Invocation)

The mechanism by which one LEZ program calls an instruction on another program within the same transaction. CPI is used to build composable programs — for example, a game program calling a token program to debit a player's balance. When using CPI via `lez-cli`, the dependency program binary is passed with `--bin-<NAME>`. The calling program must include the called program's binary in its build (typically as a `[[package.metadata.risc0.methods]]` entry).

> **🔄 Coming from Solidity?** Equivalent to an external contract call: `IToken(tokenAddr).transfer(to, amount)`. In Solidity this is a runtime call; in LEZ, the CPI is verified as part of the ZK proof of the entire instruction execution.

---

## Genesis Account

A user-created account backed by a keypair (public/private key pair). Genesis accounts can sign transactions. They are created via `wallet account new` and identified by a base58-encoded address derived from the public key. Only genesis accounts can serve as signers in `#[account(signer)]` constraints — PDAs cannot sign.

> **🔄 Coming from Solidity?** Equivalent to an EOA (Externally Owned Account) in Ethereum: an account controlled by a private key, capable of signing transactions.

---

## IDL (Interface Definition Language)

A JSON schema file describing a program's instructions, account parameters, argument types, and error codes. Generated by `lez-cli idl`, the IDL is used by `lez-cli` to call instructions, by `lez-client-gen` to produce typed client bindings, and by frontend developers to understand the program's interface. The IDL must be regenerated after any change to instruction signatures.

> **🔄 Coming from Solidity?** Equivalent to the ABI JSON file produced by Solidity compilers (`solc`). The IDL is the machine-readable interface contract for a deployed program.

---

## ImageID

The RISC Zero identifier for a compiled program guest binary. The ImageID is a cryptographic hash of the exact ELF binary produced by compiling the guest crate. It serves as the program ID: the sequencer uses it to identify which program to execute and verify for a given transaction. Because the ImageID is derived from the binary, any code change produces a different ImageID — there is no in-place upgrade mechanism.

> **🔄 Coming from Solidity?** The closest analog is the contract address in Ethereum — it identifies which code to execute. Unlike a contract address (which can be separated from its code via proxy patterns), the ImageID is a direct commitment to the code itself.

---

## lez-cli

The primary developer command-line tool for building, inspecting, deploying, and calling LEZ programs. Handles the full lifecycle: `lez-cli init` (scaffold), `lez-cli build` (compile), `lez-cli idl` (generate IDL), `lez-cli inspect` (show ImageID), `lez-cli deploy` (register on sequencer), and `lez-cli call` (invoke instructions). Most projects expose a `make cli` shorthand that wraps `lez-cli call` with pre-configured IDL and program paths.

---

## lez-client-gen

A code generator that takes a LEZ program IDL and produces typed Rust client code. The generated code includes instruction builder structs, account types, and serialization logic. With `--emit-header`, it also produces a C header for FFI consumers. Generated bindings should be committed to source control and regenerated whenever the IDL changes.

---

## lez_framework

The Rust library crate that provides the core types and traits for writing LEZ programs: `AccountWithMetadata`, `AccountPostState`, `LezError`, `LezOutput`, `LezResult`, and the `#[lez_program]` and `#[instruction]` macros. Every LEZ program's guest crate depends on `lez_framework`.

---

## LEZ

Logos Execution Zone — a specific instantiation of the LEE (Logos Execution Environment) deployed by the Logos Network. The project evolved from Nescience → NSSA (Nescience State Separation Architecture) → LSSA (Logos SSA) → LEE → LEZ. The privacy-native ZK smart contract execution layer of the Logos Network (logos.co), built by IFT (Institute of Free Technology). LEZ programs are compiled to RISC Zero guest binaries, executed in a zkVM, and verified by the `lssa` sequencer. Privacy is a first-class feature: private (shielded) accounts use commitments and nullifiers to enable confidential state transitions without revealing account contents.

---

## LezError

The error enum returned by instruction functions. Variants include framework errors (`AccountCountMismatch`, `InvalidAccountOwner`, `AccountAlreadyInitialized`, `Unauthorized`, `SerializationError`, `DeserializationError`) and user-defined errors (`Custom(u32)`). Custom error codes must start at 6000. See [Appendix C: Error Code Reference](./error-codes.md).

---

## LezOutput

The return value type from instruction functions. Wraps the post-state of all accounts, plus any events or logs the instruction emits. Constructed with `LezOutput::states_only(vec![...])` for the common case of just returning account states. The `states_only` constructor takes a `Vec<AccountPostState>` — one entry for every account that was passed in, in the same order.

---

## LezResult

A type alias: `Result<LezOutput, LezError>`. The required return type of every `#[instruction]` function. On success, return `Ok(LezOutput::states_only(...))`. On failure, return `Err(LezError::Custom(YOUR_ERROR_CODE))` or another `LezError` variant.

---

## lssa

The NSSA sequencer daemon. `lssa` is the process that receives transactions, executes and verifies LEZ program proofs, manages the commitment tree, and provides the RPC endpoint for wallets and clients. Running a local `lssa` instance is required for development and testing. `lssa` uses RocksDB for persistent state.

> **🔄 Coming from Solidity?** Analogous to a blockchain node (like `geth` or `anvil`). `lssa` is the runtime that executes your programs and maintains the canonical state.

---

## npk (Nullifier Public Key)

The SHA256 hash of the `nsk` (Nullifier Spending Key). Used to derive private account IDs: the address of a private account is deterministically derived from the `npk` and a per-account nonce. Knowing the `npk` is sufficient to identify (but not spend from) private accounts. The `npk` can be shared for address derivation without granting spending capability.

---

## nsk (Nullifier Spending Key)

The root secret for all private accounts in a LEZ wallet. All private account spending capability derives from the `nsk`. Loss of the `nsk` means permanent, unrecoverable loss of all private account contents. The `nsk` must be backed up to cold storage and never stored in plaintext on an internet-connected machine. Unlike a wallet seed phrase, the `nsk` cannot be regenerated from a mnemonic — it is a raw secret key.

---

## NSSA

Nondeterministic State Sequencer Architecture. The protocol layer underlying LEZ. NSSA defines how the sequencer manages state, commitment trees, nullifier sets, and proof verification. LEZ programs run on top of NSSA — the `lssa` daemon implements the NSSA protocol. Most application developers interact with NSSA indirectly through `lez_framework` and `lez-cli`.

---

## Nullifier

A one-way value derived from the `nsk` and a specific commitment. When a private account is "spent" (its state is consumed as input to a transaction), the corresponding nullifier is published to the sequencer's nullifier set. This prevents double-spending: the sequencer rejects any transaction that references a commitment whose nullifier has already been published. Nullifiers are unlinkable to the commitment without knowledge of the `nsk`.

> **🔄 Coming from Solidity?** Analogous to the nullifier concept in Zcash or Tornado Cash: a one-time value that marks a note as spent without revealing which note was spent.

---

## PDA (Program Derived Address)

An account address derived deterministically from seeds and a program ID, rather than from a keypair. PDAs are used for program-owned state: the program controls the PDA's data, and the PDA's address is predictable from the seeds. PDAs cannot sign transactions — only genesis (keypair) accounts can sign.

> **🔄 Coming from Solidity?** Similar to a `CREATE2` address derived from a salt and deployer address. Like a `CREATE2` contract, a PDA's address is predictable before creation, and the same seeds always produce the same address for a given program.

---

## Program

A compiled RISC Zero guest binary that implements one or more LEZ instructions. Programs are the LEZ equivalent of smart contracts: they define the logic that operates on accounts and are identified by their ImageID. Programs are stateless — all state lives in accounts passed to instructions at call time.

> **🔄 Coming from Solidity?** The program is the contract, the ImageID is the contract address, and accounts are the storage. Unlike Solidity, the program and its storage are completely separate — storage (accounts) can outlive the program that created them.

---

## Program ID

The ImageID of a deployed program. Clients and the sequencer use the program ID to identify which program to invoke for a transaction, and to verify that accounts were created/owned by the correct program. The program ID changes whenever the program code changes.

---

## RISC Zero

The zkVM technology underlying LEZ program execution. RISC Zero provides a RISC-V (specifically `riscv32im`) virtual machine whose execution can be cryptographically proven — anyone can verify that a computation was performed correctly without re-running it. LEZ programs are compiled to RISC Zero guest binaries and executed inside the RISC Zero zkVM. The `rzup` tool installs the RISC Zero toolchain.

---

## rzup

The RISC Zero toolchain installer. Installs and manages the `riscv32im-risc0-zkvm-elf` Rust target, the RISC Zero prover, and related tools. Pin the `rzup` version in your project documentation to ensure reproducible builds — different toolchain versions may produce different ImageIDs from the same source.

---

## sequencer_runner

The binary that starts and runs the `lssa` sequencer daemon. Takes a configuration directory path (not a config file path) via `--config`. Manages the RocksDB state store, listens for incoming transactions, verifies proofs, and maintains the commitment tree. For development, use `sequencer_runner --config sequencer_runner/configs/debug/`.

---

## SPEL

The Rust macro framework for writing LEZ programs. Provides `#[lez_program]`, `#[instruction]`, `#[account(...)]`, and related attribute macros. SPEL handles the boilerplate of parsing account constraints, routing instructions by name, and serializing/deserializing instruction arguments. Programs written with SPEL implement their logic as plain Rust functions decorated with `#[instruction]`.

---

## ssk (Spending Signing Key)

An ephemeral key used for a single private transaction. Unlike the `nsk` (which is a persistent root secret), the `ssk` is derived fresh for each transaction and discarded afterward. The `ssk` is used to authorize the spending of a specific private note without exposing the `nsk`. This limits the damage if a single transaction's signing material is compromised.

---

## View Tag

A 1-byte filter value derived from the viewing key and a commitment. Published alongside each commitment in the sequencer's commitment tree. Wallets first check whether a commitment's view tag matches their expected value before attempting full decryption. Because only 1/256 of commitments match any given view tag, this lets wallets skip 99.6% of commitments during scanning without missing any of their own.

---

## Visibility Mask

A per-account flag that controls what information about a private account is published when it is included in a transaction. Values:
- `0` — Public: account data and address are visible to all
- `1` — Private-authenticated: account existence is revealed but contents are hidden; only the owner (authenticated via viewing key) can decrypt
- `2` — Private-unauthenticated: account is fully hidden; scanning is required to discover membership

The visibility mask is set when creating or modifying a private account and determines the privacy level for that account's commitment.

---

## vpk (Viewing Public Key)

A secp256k1 public key derived from the `nsk`. The `vpk` enables selective disclosure: by sharing the `vpk` with a third party (e.g., an auditor), you allow them to decrypt and view private account contents without granting spending capability. The `vpk` cannot be used to authorize transactions — it is read-only access.

---

## zkVM

Zero-Knowledge Virtual Machine. A virtual machine whose execution can be cryptographically proven — specifically, that the VM executed a given program on given inputs and produced given outputs — without revealing the intermediate computation state. RISC Zero provides the zkVM underlying LEZ. The `riscv32im` instruction set architecture is the execution model inside the zkVM. ZK proofs produced by the zkVM are verified by the `lssa` sequencer for every transaction.
